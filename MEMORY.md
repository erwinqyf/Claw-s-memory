## 架构决策

### 2026-05-18: 定时任务架构重构（v2）

**旧架构问题：** 9 个独立 cron 任务，每个都启动完整 LLM session
- 每天 9+ 次 LLM 启动 → 频繁超时
- 碎片化通知 → 信息噪音
- 脚本类任务不需要 LLM

**新架构：** 脚本 + 报告分离
- **脚本层**（静默执行）：夜间脚本、ClawHub追踪、行业监控 → 结果写 data/exec-results/
- **报告层**（投递飞书）：晨间(7:00)、午间(12:00)、晚间(22:00) → 读取脚本结果，统一通知
- 通知从 9+ 条/天降到 3 条

### 2026-05-17: 子代理统筹模式

- ✅ Claw 用 `sessions_spawn` 调度子代理，结果汇总后由我汇报给丰
- 所有 cron 任务 delivery=none，静默执行

### 教训

- **deepseek API 高峰期阻塞**：00:00/06:00/23:00 最严重（deepseek-chat 和 v4-flash 均超时），07:00 近期也不稳定（2026-06-17起连续失败），09:00/11:00 正常，12:00 用 v4-flash+600s 解决
  - 解决方案：**高峰时段错开时间**比增大超时有效，00:00→04:00, 06:00→08:00, 23:00→22:00
  - 可以增加超时（600s）和换 v4-flash 的中度拥堵（12:00）：午间报告 65s
  - 23:00 也属于重度拥堵：v4-flash+600s 仍失败，移到 22:00（已验证 force-run 22:57 可成功）
- 共识翻译新范式：多引擎同时运行可减少 22% 错误（2026-05-26）
- 星巴克叫停AI库存自动盘点工具，上线9个月错误频出（2026-05-23）
- **串行队列连锁效应**：04:00 的脚本阻塞模型队列直到 ~05:00，07:00 晨间报告排队等死，08:00 ClawHub 继续排队，全链崩塌。**不要把大任务放在其他任务前 1-3h 的位置**（2026-06-17~18 教训）
- Linux VPS 可能因 DNS 导致 clawhub.ai 无法访问（2026-05-23, 2026-05-30）
- 脚本类 cron 任务超时：不要用 deepseek-v3.2，改用 qwen3.6-plus；timeout 至少 180s（2026-05-21）
- **脚本用 v4-flash，报告用 deepseek-chat**：脚本执行-夜间用 deepseek-chat 连续失败 10 次，换 v4-flash 后 17.8s 秒过。v4-flash 流更稳定。deepseek-chat 在报告类（晨间/周报复盘/行业监控）表现良好（<60s）。（2026-06-20）
