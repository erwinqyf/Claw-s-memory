## 架构决策

### 2026-05-18: 定时任务架构重构（v2）

**旧架构问题：** 9 个独立 cron 任务，每个都启动完整 LLM session，各自发飞书通知
- 每天 9+ 次 LLM 启动 → 频繁超时
- 碎片化通知 → 信息噪音
- 脚本类任务（Git同步、数据抓取）不需要 LLM

**新架构：** 脚本 + 报告分离
- **脚本层**（delivery=none）：夜间脚本、ClawHub追踪、行业监控 → 结果写 data/exec-results/
- **报告层**（delivery=announce）：晨间(7:00)、午间(12:00)、晚间(22:00) → 读取脚本结果，统一通知
- 每天从 9+ 条通知降到 3 条
- LLM session 从 9+ 降到 3-4 个

**教训：** LLM 不是万能的。纯脚本任务不需要启动完整 AI session。

### 2026-05-17: 子代理统筹模式

- ❌ 前世的多 Agent 飞书群聊方案（5 个 Bot、5 个群聊）已废弃
- ✅ 新模式：Claw 用 `sessions_spawn` 直接调度子代理，结果汇总后由我亲自汇报给丰
- 不再需要群聊、多 Bot、飞书多账号
- 所有 cron 任务 delivery=none，静默执行，我作为协调者统一向丰汇报
