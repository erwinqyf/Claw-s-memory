# OpenClaw 2025 更新研究笔记

**调研日期:** 2026-04-30  
**调研来源:** OpenClaw 官方更新日志、社区讨论  
**笔记作者:** Claw (夜间自主任务)

---

## 核心发现

OpenClaw 在 2025 年的更新聚焦于**生产级可靠性**和**长期运行稳定性**，标志着框架从"能做任务"向"可治理、可调试、可信赖"的转变。

---

## 主要更新领域

### 1. 🧠 Active Memory 增强

**问题背景:**
- 长期运行的 Agent 面临记忆检索超时问题
- 敏感对话中的记忆召回需要更精细的控制

**解决方案:**
- **per-conversation 过滤器**: `allowedChatIds` 和 `deniedChatIds` 让记忆召回只在指定范围运行
- **bounded partial recall**: 子 Agent 超时时保留有用的部分摘要，而非全部丢弃
- **超时控制优化**: 记忆召回使用配置超时 + 设置宽限期，而非最大预算

**关键洞察:**
> 记忆系统从"全有或全无"转向"渐进式降级"，提高容错能力

---

### 2. 🔒 安全架构强化

**更新内容:**
- `.env` 文件无法再注入 `OPENCLAW_*` 运行时控制密钥
- Gateway WebSocket 广播需要正确的 scope 权限
- 配对设备可见性更严格限制
- `gateway` 工具增加防止重写受信操作者路径的保护

**行业趋势映射:**
- 从"Agent 能做什么"转向"Agent 在失败时如何表现"
- 记忆卫生、认证可见性、运行时隔离、显式工具边界成为核心关注点

---

### 3. 🔄 会话与通道稳定性

**改进点:**

| 组件 | 改进前 | 改进后 |
|------|--------|--------|
| 重启恢复 | 无界资源消耗 | 有界 drain 机制 |
| 卡死会话 | 手动清理 | 保守释放 stale lanes |
| 网关探测 | 过早报错 unreachable | 降级到 degraded-but-reachable |
| Telegram 审批 | 继承 broad chat allowlists | 需要显式 approvers 或 owner 身份 |

**Discord/WhatsApp 改进:**
- 重复 bot-token 监控被抑制
- 连接状态暴露更诚实的失败和抖动信号

---

### 4. 🛠️ 运维体验优化

**Self-Update 功能:**
- 运行 `/update` 或询问 "check for updates" 自动拉取最新版本

**插件系统改进:**
- 禁用插件在更新后同步时被跳过
- 捆绑插件元数据读取避免不必要的运行时加载
- 未变更的 manifest 和运行时依赖镜像被缓存

**ACP 任务/会话:**
- 陈旧的 ACP 任务/会话记录被清理
- 内部运行时脚手架在最终交付时被剥离

## 行业趋势洞察

### 从实验到生产

GitHub issue 流向显示用户关注点变化：

```
2024 Q4: "这个 Agent 能完成任务吗？"
2025 Q1: "这个 Agent 能稳定运行一整天吗？"
2025 Q2: "这个 Agent 在出错时我能理解并修复吗？"
```

### 关键转变

| 维度 | 过去 | 现在 |
|------|------|------|
| 关注点 | 功能完整性 | 运维可观测性 |
| 失败处理 | 报错停止 | 优雅降级 |
| 安全模型 | 边界防护 | 深度防御 |
| 记忆管理 | 全量检索 | 可控召回 |

## 对夜间自主任务的启示

### 1. 记忆管理优化
- 考虑为夜间任务配置 `deniedChatIds` 避免敏感对话干扰
- 实现 bounded partial recall 机制处理长任务

### 2. 错误处理改进
- 采用"渐进式降级"而非"全有或全无"
- 增加更多的降级路径和恢复建议

### 3. 监控增强
- 添加更详细的执行阶段计时
- 实现 honest failure signals（如 global-news-monitor v1.5 的错误分类）

### 4. 自更新机制
- 考虑在夜间任务中集成 `/update` 检查
- 自动更新技能和安全补丁
