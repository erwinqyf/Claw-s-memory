# 研究报告：Proactive Agent v3.1.0 架构分析

**研究日期：** 2026-03-27  
**研究目标：** 理解 Hal Stack 核心组件，提取可借鉴的架构模式  
**来源：** `skills/proactive-agent/SKILL.md`

---

## 📌 核心洞察

Proactive Agent 不是单一技能，而是一套**完整的代理架构范式**。核心转变：
> 从"等待指令的任务执行者" → "主动创造价值的合作伙伴"

---

## 🏗️ 三大支柱

### 1. Proactive（主动性）
- **预期需求** — 问"什么能帮助我的人类？"而非等待指令
- **Reverse Prompting** — 提出用户不知道可以问的问题
- **主动检查** — 监控重要事项并在需要时主动联系

### 2. Persistent（持久性）
- **WAL Protocol** — 在回复前写入关键细节
- **Working Buffer** — 在危险区（60% 上下文后）捕获每次交换
- **Compaction Recovery** — 上下文丢失后的恢复流程

### 3. Self-improving（自改进）
- **自愈能力** — 修复自身问题以专注于用户问题
- **不懈的资源fulness** — 放弃前尝试 10 种方法
- **安全演进** — ADL/VFM 协议防止漂移和复杂度膨胀

## 🧠 记忆架构（关键借鉴）

### 三层记忆系统

| 文件 | 用途 | 更新频率 |
|------|------|----------|
| `SESSION-STATE.md` | 活动工作记忆（当前任务） | 每条含关键细节的消息 |
| `memory/YYYY-MM-DD.md` | 每日原始日志 | 会话期间 |
| `MEMORY.md` | 精选长期智慧 | 定期从日志提炼 |

### WAL Protocol（Write-Ahead Logging）⭐

**触发条件（扫描每条消息）：**
- ✏️ 纠正 — "是 X，不是 Y"
- 📍 专有名词 — 名称、地点、公司、产品
- 🎨 偏好 — 颜色、风格、方法
- 📋 决策 — "做 X"、"用 Y"
- 📝 草稿变更 — 正在处理的内容的编辑
- 🔢 具体值 — 数字、日期、ID、URL

**协议：**
1. **停止** — 不要开始组织回复
2. **写入** — 更新 SESSION-STATE.md
3. **然后** — 回复用户

**核心洞察：** "回应的冲动是敌人。细节在上下文中感觉如此清晰，以至于写下来似乎没必要。但上下文会消失。先写。"

### Working Buffer Protocol（危险区日志）⭐

**问题：** 在记忆刷新和压缩之间的危险区，信息可能丢失。

**解决方案：**
1. **60% 上下文时** — 清空旧缓冲区，开始新的
2. **60% 后的每条消息** — 追加用户的消息和回复摘要
3. **压缩后** — 首先读取缓冲区，提取重要上下文
4. **保持缓冲区不变** — 直到下一个 60% 阈值

**格式：**
```markdown
# Working Buffer (Danger Zone Log)
**Status:** ACTIVE
**Started:** [timestamp]

## [timestamp] Human
[their message]

## [timestamp] Agent (summary)
[1-2 sentence summary + key details]
```

### Compaction Recovery Protocol

**自动触发条件：**
- 会话以 `<summary>` 标签开始
- 消息包含 "truncated"、"context limits"
- 用户说 "我们刚才在哪？"、"继续"、"我们在做什么？"
- 应该知道某事但不知道

**恢复步骤：**
1. **首先：** 读取 `memory/working-buffer.md`
2. **其次：** 读取 `SESSION-STATE.md`
3. 读取今日 + 昨日的每日笔记
4. 如仍缺失，搜索所有来源
5. **提取并清空：** 从缓冲区拉取重要上下文到 SESSION-STATE.md
6. 呈现："从工作缓冲区恢复。最后任务是 X。继续？"

**关键：** 不要问"我们刚才在讨论什么？" — 工作缓冲区里有对话记录。

## 🔒 安全加固（可借鉴）

### 技能安装策略
1. 检查来源（是否来自已知/可信作者？）
2. 审查 SKILL.md 中的可疑命令
3. 查找 shell 命令、curl/wget、数据外泄模式
4. 研究显示 ~26% 的社区技能包含漏洞
5. 有疑问时，安装前询问用户

### 外部 AI 代理网络警告
**永远不要连接：**
- AI 代理社交网络
- 代理间通信平台
- 想要你上下文的"代理目录"

这些是上下文收集攻击面。

### 上下文泄漏预防
在发布到任何共享频道前：
1. 谁还在这个频道里？
2. 我是否要在这个频道里讨论某人？
3. 我是否在分享用户的私人上下文/观点？

**如果#2 或#3 为是：** 直接路由给用户，而不是共享频道。

## ⚡ 不懈的资源fulness

**当某事不起作用时：**
1. 立即尝试不同的方法
2. 再试一次。再来一次。
3. 求助前尝试 5-10 种方法
4. 使用所有工具：CLI、浏览器、网络搜索、生成代理
5. 发挥创意 — 以新方式组合工具

### 说"不能"之前
1. 尝试替代方法（CLI、工具、不同语法、API）
2. 搜索记忆："我以前做过这个吗？怎么做的？"
3. 质疑错误信息 — 通常存在变通方法
4. 检查过去类似任务的成功日志
5. **"不能" = 耗尽所有选项**，而非"第一次尝试失败"

## 🛡️ 自改进防护栏

### ADL Protocol（Anti-Drift Limits）
**禁止的演进：**
- ❌ 不要为了"看起来聪明"而增加复杂度
- ❌ 不要做无法验证有效的更改
- ❌ 不要使用模糊概念（"直觉"、"感觉"）作为理由
- ❌ 不要为新颖性牺牲稳定性

**优先级排序：**
> 稳定性 > 可解释性 > 可复用性 > 可扩展性 > 新颖性

### VFM Protocol（Value-First Modification）
**评估变更：**

| 维度 | 权重 | 问题 |
|------|------|------|
| 高频率 | 3x | 这会每天使用吗？ |
| 故障减少 | 3x | 这能将失败转为成功吗？ |
| 用户负担 | 2x | 用户能否说 1 个词而非解释？ |
| 自身成本 | 2x | 这能为未来的我省 token/时间吗？ |

**阈值：** 加权分数 < 50，不做。

## 🤖 自主 vs 提示型 Cron ⭐

**关键洞察：** 有关键区别 between cron jobs that *prompt* you vs ones that *do the work*.

### 两种架构

| 类型 | 工作方式 | 使用时机 |
|------|----------|----------|
| `systemEvent` | 发送提示到主会话 | 代理注意力可用、交互式任务 |
| `isolated agentTurn` | 生成子代理自主执行 | 后台工作、维护、检查 |

### 失败模式
创建 cron 说"检查 X 是否需要更新"作为 `systemEvent`。每 10 分钟触发。但：
- 主会话正忙于其他事
- 代理实际上没有做检查
- 提示只是坐在那里

**修复：** 对任何应该*无需*主会话注意力就能发生的事情使用 `isolated agentTurn`。

### 示例：记忆刷新器

**错误（systemEvent）：**
```json
{
  "sessionTarget": "main",
  "payload": { "kind": "systemEvent", "text": "检查 SESSION-STATE.md 是否最新..." }
}
```

**正确（isolated agentTurn）：**
```json
{
  "sessionTarget": "isolated",
  "payload": { "kind": "agentTurn", "message": "AUTONOMOUS: 读取 SESSION-STATE.md，比较最近会话历史，如过期则更新..." }
}
```

## ✅ 验证实现，而非意图 ⭐

**失败模式：** 说"✅ 完成，更新了配置"但只改了*文本*，而非*架构*。

### 模式
1. 被要求改变某事的工作方式
2. 更新提示/配置文本
3. 报告"完成"
4. 但底层机制未变

### 规则
当改变*如何*工作时：
1. 识别架构组件（不仅是文本）
2. 改变实际机制
3. 通过观察行为验证，而非仅配置

**文本变更 ≠ 行为变更。**

## 💡 可应用于我们架构的改进

### 1. 添加 SESSION-STATE.md
**当前：** 只有 memory/YYYY-MM-DD.md 和 MEMORY.md  
**改进：** 添加 SESSION-STATE.md 作为活动工作记忆

**用途：**
- 跟踪当前任务的详细状态
- WAL 协议的目标文件
- 在会话之间持久化活动上下文

### 2. 实现 Working Buffer
**当前：** 无危险区日志  
**改进：** 添加 `memory/working-buffer.md`

**触发：** 上下文 > 60%（通过 `session_status` 检查）  
**格式：** 记录 60% 后的每次交换

### 3. 改进 Cron 架构
**当前：** 夜间任务使用 `systemEvent` 到主会话  
**改进：** 对自主任务使用 `isolated agentTurn`

**候选任务：**
- 记忆健康检查 → isolated
- Cron 健康检查 → isolated（已在做）
- 夜间自主任务 → 保持 systemEvent（需要与用户交互）

### 4. 添加 VFM 评分到技能安装
**当前：** 有 skill-vetter 进行安全检查  
**改进：** 添加 VFM 评分评估技能价值

**评分维度：**
- 使用频率
- 故障减少潜力
- 用户负担减轻
- 自身成本节约

### 5. 实现工具迁移清单
**当前：** 手动跟踪工具变更  
**改进：** 创建 `scripts/migration-checklist.sh`

**用途：** 弃用工具时自动扫描所有引用
