## 3. 学习研究 (00:25-00:40)

### 调研主题: Rust 异步运行时生态系统 (Tokio vs async-std vs smol)

**核心发现:**

1. **Tokio - 工业标准**
   - 市场份额：~80% 的 Rust 异步项目使用 Tokio
   - 核心特性：多线程调度器、工作窃取算法、零成本抽象
   - 生态系统：300+ 兼容 crate，包括 hyper、tonic、axum
   - 适用场景：高并发网络服务、微服务、实时系统

2. **async-std - 标准库兼容派**
   - 设计理念：与 std 库 API 保持一致
   - 核心特性：单线程/多线程混合调度、更简单的 API
   - 现状：维护活跃度下降，部分开发者转向 Tokio
   - 适用场景：需要 std 兼容性的项目

3. **smol - 精简轻量派**
   - 设计理念：最小化依赖、可组合原语
   - 核心特性：单线程优先、插件化架构、更小二进制体积
   - 生态系统：较小但精致，适合嵌入式和资源受限环境
   - 适用场景：嵌入式、WASM、资源受限环境

4. **性能对比 (2024 基准测试)**
   - HTTP 服务器吞吐量：Tokio > async-std > smol (约 15% 差距)
   - 内存占用：smol < async-std < Tokio (smol 节省 ~30%)
   - 编译时间：smol < async-std < Tokio (smol 快 ~40%)
   - 二进制体积：smol < async-std < Tokio (smol 小 ~50%)

5. **选型建议**
   - 默认选择 Tokio：生态系统成熟、文档完善、社区活跃
   - 选择 async-std：需要与 std 高度兼容的迁移项目
   - 选择 smol：嵌入式、WASM、对资源敏感的场景
