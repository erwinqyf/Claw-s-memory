# Sub-Agent 分工体系 v2

> 基于第一性原理重构，2026-05-18  
> 核心思想：LLM 只用在真正需要智能的地方

## 架构

```
丰 (人类)
  ↑
  │  飞书通知（3次/天：7:00 / 12:00 / 22:00）
  │
Claw 🪞 (协调者 / Orchestrator)
  │
  ├── 📊 脚本执行层（delivery=none，静默执行）
  │   ├── 🔧 夜间脚本 (00:00) → deepseek-v3.2
  │   ├── 🔍 ClawHub追踪 (06:00) → qwen3.6-plus
  │   ├── 🔍 行业监控 (二/四/六 11:00) → glm-5
  │   └── 🔧 小红书周报 (周五 21:00) → deepseek-v3.2
  │
  └── 📋 报告层（delivery=announce，统一通知）
      ├── 📋 晨间报告 (07:00) → MiniMax-M2.5
      ├── 📰 午间报告 (12:00) → MiniMax-M2.5
      ├── 📋 晚间报告 (22:00) → MiniMax-M2.5
      ├── 📋 周报复盘 (周一 09:00) → MiniMax-M2.5
      └── 📋 每周记忆巩固 (周日 03:00) → MiniMax-M2.5
```

## 模型分配

| 模型 | 分配角色 | 原因 |
|------|---------|------|
| **deepseek-v3.2** | 🔧 Engineer | 代码能力强，脚本执行/调试最优 |
| **glm-5** | 🔍 Researcher | 知识问答强，行业分析/数据解读 |
| **qwen3.6-plus** | 🔧 全能兜底 | 支持 vision，全能型，稳定性高 |
| **MiniMax-M2.5** | 📋 Reporter | 中文写作流畅，报告/周报质量好 |

## 设计原则

1. **脚本与报告分离** — 脚本任务只执行、不通知；报告任务只读结果、做汇总
2. **数据驱动** — 脚本结果写入 `data/exec-results/`，报告从中读取
3. **通知聚合** — 每天只 3 条飞书通知（晨/午/晚），不碎片化
4. **精简 prompt** — 每个任务指令 < 150 字，减少超时风险
5. **模型按需分配** — 不同角色用最适合的模型
6. **LLM 用在刀刃上** — 只有需要理解、分析、写作的任务才启动 LLM

## 数据流

```
脚本执行 → data/exec-results/*.json → 报告读取 → 飞书通知
```

## 失败处理

- 脚本失败：报告层读取时标注"执行失败"，通知丰
- 报告失败：单独 cron 不互相依赖
